# status_2026-02-10_2148

## 요약

- 현재 레포(`C:\github\hwp-mcp`)에서 HWP/HWPX를 **한글(HWP) COM 자동화**로 열고, 한글의 **원래 “다른 이름으로 저장(HTML)” 기능(SaveAs)**에 의존해 HTML을 생성하는 흐름은 동작한다.
- 다만 “페이지 레이아웃(페이지 모양)”이 **어떤 저장 경로/옵션으로 저장되느냐**에 따라 브라우저에서 다르게 보일 수 있으며, 최근 GUI/스크립트 실행에서 사용자가 기대하는 “페이지 모양”이 재현되지 않는 문제가 발생했다.
- 코드에서 임의로 CSS를 “주입”해서 페이지처럼 보이게 만드는 접근은 **사용자 요청으로 철회**했다(이 문서 작성 시점 기준, 주입 코드는 제거됨).

## 현상(사용자 보고)

- GUI 스크립트로 변환 시 HTML은 생성되지만:
  - 이미지가 빠져 보이거나
  - (핵심) **페이지 레이아웃 자체가 없어 보이고** “원래 제대로 나오던 HTML”과 다르게 보임

## 확인된 사실(관찰)

### 1) MCP 유무와 무관하게 변환 가능

- `convert_hwpx_to_html.py` / `convert_hwpx_to_html_gui.py`는 MCP를 호출하지 않고, 파이썬이 직접 HWP COM을 호출하므로 **Cursor/MCP 없이도 실행 가능**하다.
- MCP는 Cursor에서 tool 호출을 하기 위한 통로이며, 변환의 본질은 **HWP COM 자동화**이다.

### 2) “보안 경고(모두 허용)” 다이얼로그가 자동화를 막을 수 있음

- 한글이 파일 접근/저장 시 보안 경고를 띄울 수 있고, 이 모달이 떠 있으면 COM 호출이 멈춘 것처럼 보인다.
- 이를 완화하기 위해 `HwpController.connect()` 시점에 UIA 기반으로 “모두 허용”을 자동 클릭하는 워처를 추가했고, `requirements.txt`에 `pywinauto`를 추가했다.

### 3) HTML 내 이미지 참조가 `file:///C:\Temp\...`로 나오는 케이스가 있음

- 한글의 HTML 내보내기 결과가 이미지 리소스를 임시 폴더(C:\Temp 등)로 떨어뜨리고,
  HTML은 그 경로를 `file:///...`로 참조하는 케이스가 관찰됐다.
- 이를 해결하려고 HTML 내 `src="file:///..."`를 같은 폴더의 `<문서명>_files/`로 치환하고, 실제 이미지 파일도 복사하는 후처리를 추가했다.
- 이 후처리는 “페이지 레이아웃”과 직접적 관련이 없으며, **리소스 참조를 로컬화**하기 위한 목적이다.

## 문제점(핵심)

- 사용자가 기대하는 “페이지 모양(레이아웃)”이 **어떤 HTML 저장 형식/옵션**으로 생성되었는지 명확하지 않다.
- 단순 `SaveAs(..., "HTML", "")` 결과가 사용자가 보던 “페이지 레이아웃이 있는 HTML”과 동일하지 않을 수 있다.
  - 한글 UI의 [다른 이름으로 저장]에는 “인터넷 문서(*.htm)” 외에 “인터넷 프레젠테이션문서(*.htm)” 같은 별도 형식이 존재(한컴 도움말에 명시).
  - 하지만 COM `SaveAs`의 format 문자열로 이 형식을 안정적으로 선택하는 방법은 아직 확정되지 않았다.

## 원칙(사용자 방향)

- “페이지 레이아웃을 억지로 재현하기 위한 CSS/HTML 주입” 같은 임의 변형은 하지 않는다.
- 해결 방향은 **한컴(HWP) 프로그램이 제공하는 ‘원래 HTML 저장 기능’**(=UI에서 선택 가능한 올바른 형식/옵션)을 자동화로 **정확히 호출/선택**하는 것이다.
